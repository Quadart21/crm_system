**Type**: app

**Status**: active working version

## What It Does

- Показывает премиальный интерфейс Rick CRM с тёмной/светлой темой, ролевой навигацией и модальными рабочими сценариями.
- На стартовом экране входа показывает крупный кликабельный Telegram-контакт `TG: @MIRAA_BOOM` с подписью `ЖДУ СВЯЗИ`, ссылка ведёт на `https://t.me/miraa_boom`.
- Ведёт сотрудников и роли: супер-админ, админ, саппорт, курьер.
- Создаёт города и автоматически заполняет 15 базовых товаров с расценками.
- Настройки разделены на вкладки `Города`, `МП`, `Бонусы`, `Статусы курьера`, `Типы проблем`, `Решения диспутов`; raw JSON из UI настроек убран; в редактируемых справочниках добавлены добавление/редактирование/удаление с подтверждением.
- В `Настройки → Статусы курьера` можно добавлять, редактировать и удалять статусы; прежние дефолтные статусы (`Зелёный`, `Жёлтый`, `Оранжевый`, `Красный`, `Критический`) автоматически восстанавливаются без удаления пользовательских статусов и доступны кнопкой `Восстановить прежние`.
- В `Настройки → Статусы курьера` логика диспутов редактируется конструктором: для `Возврат` и `Купон` отдельно задаются чекбоксы компонентов списания (`ничего`, `себестоимость`, `работа`, `розница`, проценты, `полный ущерб`, `ручной режим`) и процентные поля.
- В настройках города супер-админ редактирует город в расширенном окне, переименовывает товар, включает/выключает товар по режимам `Розница`/`Партии`, меняет таблицу `Вес → Цена за работу` с увеличенными полями ввода и отдельные надбавки `Тип-1`/`Тип-2`/`Тип-3`.
- Управляет партиями через модалки: создание, выдача одному активному курьеру, `Цена товара`, необязательная `Оплата работы склада`, автоматически рассчитанная реальная себестоимость 1 г, отдельный ФАС, остаток в граммах, закрытие партии при потере; клик по выданной партии открывает расчёты расхода, заработка, остатка и стоимости товара на руках.
- В разделе `Партии` доступны фильтры по статусу, товару, курьеру, городу и дате.
- Принимает и редактирует ввод данных по партии: курьер, город, несколько строк `вес + тип + количество + распределение по МП`, видимый расчёт заработка до сохранения; при редактировании пересчитывает остаток и финансовую строку.
- Создаёт финансовые строки «в обработке» и даёт распределять заработок в «на руки», «залог», «банк» с понятными подписями и строгим правилом: сумма трёх полей должна ровно равняться заработку ввода; обработанные вводы атомарно накапливаются в общем доходе сотрудника.
- Открывает тикеты/диспуты через форму с выбором курьера и выданной партии; у тикета/диспута есть свободная метка `Номер заказа`, а конкретный ввод данных CRM подбирает автоматически по свободному адресу в статистике и резервирует в `DataEntryIssueAllocation`.
- При закрытии диспута показывает предварительный `Расчёт списания`, сохраняет решение/coupon/calculationData, но не списывает деньги сразу; финальное списание выполняется при выплате банка после ввода продаж за период.
- Ведёт банк, залог, штрафы, историю финансов и логи действий; выплата банка стала финальным расчётом периода: продажи → % диспутов от продаж → статус курьера → списания закрытых диспутов → бонус → выплата.
- Дашборд стал объёмным аналитическим экраном: быстрые периоды `7 дней` / `30 дней` / `Месяц` / `Всё`, ручной период, группировка День / Неделя / Месяц, фильтр по МП, общая сводка, динамика, маркетплейсы по отдельности, курьеры, ввод данных/партии, финансы и диспуты.
- Ограничивает видимость по ролям: курьер видит только свою статистику и без МП; строки курьеров в статистике открывают подробные карточки с партиями, тикетами/диспутами, заработком, балансами и статусом.

## Data Stored

- **Employee**: внутренние пользователи CRM, роли, статус, балансы.
- **City**: города CRM.
- **Product**: 15 базовых товаров.
- **CityProductSetting**: активность товара в городе и режиме.
- **PriceRate**: цены по городу, товару, режиму и весовке.
- **StashType**: типы клада и надбавки отдельно для розницы и партий.
- **Marketplace**: МП, видимые всем кроме курьера.
- **Batch**: партии, вес, цена товара (`totalBatchCost`), оплата работы склада (`warehouseWorkCost`, по умолчанию `0`), производная реальная себестоимость 1 г (`costPerGram`), отдельный ФАС, остаток, курьер, статус.
- **DataEntry**: ввод данных по адресам без отдельной сущности адреса.
- **IssueRecord**: тикеты и диспуты.
- **DataEntryIssueAllocation**: резерв адреса из ввода данных под конкретный тикет/диспут.
- **Penalty**: штрафы и списания из банка/залога.
- **FinanceProcessing**: строки финансов «в обработке».
- **FinanceLedger**: история выплат/начислений/списаний.
- **MonthlySettlement**: слепок финального расчёта выплаты банка за период по курьеру.
- **SettlementDisputeWriteOff**: финальные списания по закрытым диспутам внутри выплаты банка, с защитой от повторного списания.
- **ProblemType**: типы проблем.
- **DisputeDecision**: варианты решения диспута.
- **CourierStatusRule**: правила статусов курьера.
- **AppSetting**: настройки лимитов и бонусов.
- **AppLog**: неизменяемые логи действий, кроме полной очистки супер-админом.

## Functions

- `getAppState()`: создаёт seed при первом запуске и возвращает все данные приложения.
- `getDashboardAnalytics(...)`: агрегирует дашборд по периоду, группировке и МП: summary, timeline, marketplaceRows, courierRows, finance, disputes, dataQuality.
- `loginEmployee({ login, password })`: внутренняя авторизация сотрудника CRM.
- `createCity(...)`: создаёт город и заполняет 15 товаров/расценки.
- `createEmployee(...)`, `updateEmployee(...)`: управление сотрудниками.
- `createBatch(...)`, `issueBatch(...)`, `closeBatch(...)`: цикл жизни партии; `issueBatch` принимает цену товара, ФАС и необязательную оплату работы склада, затем сам считает реальную себестоимость `costPerGram = (totalBatchCost + включённый fasCost + warehouseWorkCost) / weight`.
- `createDataEntry(...)`, `editDataEntry(...)`: ввод/редактирование адресов и пересчёт финансовой строки.
- `calculateFinance(...)`: строгое распределение заработка по на руки/залог/банк ровно на сумму ввода; использует атомарные increment-обновления балансов и защищает строку от повторной обработки.
- `openIssue(...)`, `previewIssueAllocation(...)`, `closeTicket(...)`, `closeDispute(...)`: тикеты и диспуты; `openIssue` сохраняет `orderNumber`, автоматически подбирает свободный `DataEntry` по партии/параметрам и резервирует адрес, `closeDispute` закрывает диспут без финального списания.
- `applyPenalty(...)`: штраф со списанием сначала из банка, затем из залога.
- `previewBankPayout(...)`, `payoutBank(...)`: выплата банка с периодом и продажами, расчётом статуса от всех диспутов периода, блокировкой при открытых диспутах >10%, финальными списаниями закрытых диспутов и settlement-историей.
- `upsertMarketplace(...)`, `updateCity(...)`, `updateProduct(...)`, `updateCityProductSetting(...)`, `updatePriceRate(...)`, `updateStashType(...)`, `createCourierStatusRule(...)`, `updateCourierStatusRule(...)`, `restoreDefaultCourierStatusRules(...)`, `updateHandLimitRules(...)`, `updateBonusRules(...)`, `addProblemType(...)`, `updateProblemType(...)`, `addDisputeDecision(...)`, `updateDisputeDecision(...)`: справочники и настройки через формы.
- `deleteCity(...)`, `deleteProduct(...)`, `deleteMarketplace(...)`, `deletePriceRate(...)`, `deleteStashType(...)`, `deleteCourierStatusRule(...)`, `deleteProblemType(...)`, `deleteDisputeDecision(...)`: удаления настроек с подтверждением на UI и безопасной логикой на backend; города/товары/МП с историей архивируются отключением, типы проблем и решения диспутов с историческими связями блокируются, строки цен/надбавки удаляются физически.
- `clearLogs(...)`: полная очистка логов супер-админом с новой записью об очистке.

## Known Limitations / Follow-ups

- Дашборд всё ещё использует схематичный визуальный график; клиентские показатели фильтруются по периоду, но следующий шаг — заменить график на полноценные временные серии.
- Пароли сотрудников хранятся в текущей MVP-схеме как строки для соответствия внутреннему логину/паролю из ТЗ; перед реальной эксплуатацией стоит заменить на хэширование.
- Backend применяет ролевые проверки по `actorLogin`; это подходит для внутреннего тестового логина CRM, но перед боевой эксплуатацией нужно перейти на полноценную сессионную авторизацию.

## Use Cases

- Рабочее тестирование Rick CRM по основным потокам: партия → выдача → ввод/редактирование данных → финансы → тикеты/диспуты → штрафы/выплаты → логи.
- Демонстрация ролевого интерфейса: супер-админ, админ, саппорт, курьер.
- База для следующей итерации по строгому ТЗ: детальные ценовые настройки и полноценные графики.
